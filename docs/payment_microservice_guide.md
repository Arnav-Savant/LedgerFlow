## Service Overview

The payment service is the financial execution layer of the LedgerFlow platform. It owns everything related to *how* money moves: initiating payment captures with external providers, tracking payment state transitions, and publishing the outcome events that allow commerce, ledger, and notification services to react.

Its boundary starts where commerce ends. When commerce creates a checkout and its orders are ready for payment, commerce publishes a `CheckoutPaymentRequested` event. The payment service consumes this event, coordinates with an external payment provider (e.g. Razorpay, Stripe), and publishes `PaymentCaptured` or `PaymentFailed` back to the platform. It does not know about products, sellers, inventory, or order line items â€” that domain belongs entirely to commerce.

The payment service is the sole owner of `Payment` and `Refund` records. No other service writes to these tables.

## Architectural Overview

The service follows the same layered architecture as the commerce service, with strict unidirectional dependency flow:

```
HTTP layer (routes)  â†’  Service layer  â†’  Repository layer  â†’  Database
        â†‘                     â†‘
  Middleware             Domain models
  Schemas                Enums / Exceptions
                         Response envelopes
```

All design principles from the commerce service apply here without exception. The rules are repeated below for completeness; do not treat any of them as optional.

**Key design principles in effect:**

- **No business logic in repositories.** Repos are purely data access. State transitions and multi-step operations belong in the service layer.
- **Every layer has try/except.** Repository methods catch `SQLAlchemyError` â†’ re-raise as `DatabaseException`. Service methods catch `AppException` subclasses and re-raise unchanged; catch bare `Exception` â†’ wrap as `ServiceException`. Route handlers catch `AppException` â†’ return `ErrorResponse` with correct status code; catch bare `Exception` â†’ return 500 `ErrorResponse`.
- **Repos flush, services commit.** `db.flush()` in repositories. `db.commit()` / `db.rollback()` owned exclusively by the service layer. This keeps multi-step operations atomic without compensating transactions.
- **Services own object instantiation via `__init__`.** Each service creates its own repository instances in `__init__`. No module-level singleton repos. Routes create service instances inside each handler function.
- **Service layer owns cross-domain coordination through services only.** A service may call other services but must never directly access a repository owned by another domain.
- **Routes catch, not propagate.** Route handlers do not let exceptions bubble to the global exception handler. The global handlers in `main.py` are a safety net only.
- **Singleton configuration objects.** `server_config` and `postgres_config` are module-level instances. `extra="ignore"` prevents cross-config contamination.
- **Singleton logger.** `AppLogger` uses `__new__` to enforce a single instance.
- **IDs as UUID strings.** All PKs and FKs are `VARCHAR(36)` storing standard UUID strings generated by the application via `str(uuid.uuid4())`.
- **Money as integers.** All monetary amounts stored in smallest denomination (paise for INR). No floats for money anywhere.
- **PostgreSQL enum types owned by Alembic.** All `ENUM` types created via `op.execute("CREATE TYPE ...")` in the migration. SQLAlchemy models use `create_type=False`.

## Project Structure

### `config/`

Identical in structure and purpose to the commerce service's `config/`. Nothing outside `config/` reads environment variables directly.

- `server_config.py` â€” `ServerConfig` reads `APP_*` prefixed variables. Service name is `payment-service`, default port `8002`.
- `postgres_config.py` â€” `PostgresConfig` reads `POSTGRES_*` prefixed variables. Exposes `sync_url` and `async_url`. Only `sync_url` is used currently.
- `database.py` â€” Creates the SQLAlchemy `engine` and `SessionLocal` factory. Exposes `get_db()` as a FastAPI dependency.
- `logger.py` â€” `AppLogger` singleton. Log lines follow the format `timestamp | LEVEL | payment-service | message | key=value ...`.

### `models/`

- `base.py` â€” `Base` (declarative base) and `TimestampMixin` (`created_at`, `updated_at`). `onupdate` is ORM-layer only â€” raw SQL `UPDATE` statements must manually include `updated_at = now()`.
- `__init__.py` â€” Will import all model classes once they are defined. This file is imported as a side effect in `migrations/env.py` to register models with `Base.metadata`.

**Domain models to implement:**

- `Payment` â€” Represents one payment attempt for a checkout. Fields: `id`, `checkout_id` (FK to commerce, stored as VARCHAR, no DB-level FK constraint since it's cross-service), `amount`, `currency`, `status` (`PaymentStatus`), `payment_method` (`PaymentMethod`), `provider_reference` (external gateway transaction ID), `failure_reason`, `created_at`, `updated_at`.
- `Refund` â€” Represents a refund against a captured payment. Fields: `id`, `payment_id` (FK to Payment), `amount`, `currency`, `status` (`RefundStatus`), `provider_reference`, `reason`, `created_at`, `updated_at`.

### `repository/`

One repository class per entity. Each accepts `Session` as the first argument to every method. All mutation methods use `db.flush()`. Every method wraps its body in `try/except SQLAlchemyError` â†’ re-raises as `DatabaseException`.

**Repositories to implement:**
- `PaymentRepo` â€” CRUD + `update_status(payment_id, status, failure_reason=None)`, `get_by_checkout_id(checkout_id)`.
- `RefundRepo` â€” CRUD + `get_all_by_payment_id(payment_id)`, `update_status(refund_id, status)`.

### `migrations/`

- `env.py` â€” Alembic runtime. Imports `postgres_config` to override the DB URL and `import models` as side effect to populate `Base.metadata`.
- `alembic.ini` â€” Sets `script_location = migrations`. URL is left blank; injected by `env.py`.

**First migration to create:** `0001_initial_schema.py` â€” will create `payment_status`, `payment_method`, `refund_status` PostgreSQL enum types and the `payments` and `refunds` tables.

### `seeders/`

No seed data is planned for the payment service. The `payments` and `refunds` tables are populated exclusively by live request flows and event consumption. The `seeders/` folder exists for structural consistency and potential future use.

### `utils/`

#### `utils/enums.py`

Python enum classes mirroring PostgreSQL enum types:

- `PaymentStatus` â€” `INITIATED`, `PENDING`, `CAPTURED`, `FAILED`, `REFUND_INITIATED`, `REFUNDED`, `CANCELLED`
- `PaymentMethod` â€” `UPI`, `CARD`, `NET_BANKING`, `WALLET`
- `Currency` â€” `INR`, `USD`, `GBP`, `EUR`, `JPY`
- `RefundStatus` â€” `INITIATED`, `PROCESSED`, `FAILED`

#### `utils/common/`

Identical contracts to the commerce service:

- `custom_exception.py` â€” `AppException` base class and subclasses: `NotFoundException`, `ConflictException`, `ValidationException`, `DatabaseException`, `ServiceException`, `PaymentException` (HTTP 402, error code `PAYMENT_FAILED` â€” raised when a provider rejects or a gateway error occurs).
- `success_response.py` â€” `SuccessResponse` with `ok()` and `created()` classmethods.
- `error_response.py` â€” `ErrorResponse` with `from_exception()` and `internal_error()` classmethods.

`PaymentException` is the one addition beyond the commerce exception hierarchy. It maps to HTTP 402 and signals provider-level payment failure, invalid payment state, or gateway rejection.

### `services/`

Note: This service uses the directory name `services/` (plural) rather than `service/` (singular) as in the commerce service. Keep this consistent â€” do not rename.

Business logic layer. The same instantiation and exception patterns apply as in the commerce service.

**Services to implement:**

- `payment_service.py` â€” Owns the payment lifecycle. Creates `PaymentRepo` in `__init__`. Methods: `initiate_payment(checkout_id, amount, currency, payment_method)` â†’ creates `Payment` record, calls provider stub, commits. `capture_payment(payment_id)` â†’ updates status to `CAPTURED`. `fail_payment(payment_id, reason)` â†’ updates status to `FAILED`. `get_by_id(payment_id)` â†’ fetch enriched payment. `get_by_checkout_id(checkout_id)` â†’ fetch payment for a checkout.
- `refund_service.py` â€” Owns refund lifecycle. Creates `RefundRepo` and uses `PaymentService` in `__init__`. Methods: `initiate_refund(payment_id, amount, reason)` â†’ validates payment is `CAPTURED`, creates `Refund`, updates payment to `REFUND_INITIATED`. `process_refund(refund_id)` â†’ updates refund to `PROCESSED`. `fail_refund(refund_id)` â†’ updates refund to `FAILED`.

Transaction ownership: `payment_service` and `refund_service` call `db.commit()` / `db.rollback()` for their own top-level operations. Sub-service calls (if any) flush only.

### `schema/`

Pydantic request/response models. To be defined when routes are implemented.

**Planned schemas:**
- `payment_schema.py` â€” `PaymentInitiateRequest`, `PaymentResponse`, `PaymentDetailResponse`.
- `refund_schema.py` â€” `RefundInitiateRequest`, `RefundResponse`.

### `routes/`

FastAPI route handlers. Every handler owns its own try/except and returns structured `SuccessResponse` / `ErrorResponse`. No exceptions propagate to the global handler during normal request processing.

**Planned routes:**
- `payment_routes.py` â€” `POST /payments/initiate`, `GET /payments/{payment_id}`, `GET /payments/checkout/{checkout_id}`.
- `refund_routes.py` â€” `POST /refunds/initiate`, `GET /refunds/{refund_id}`.

### `middleware/`

No middleware is implemented yet. Future middleware may handle idempotency key enforcement for payment initiation requests.

### `main.py`

FastAPI app entry point. Lifespan hook runs migrations before the server accepts traffic (no seeders â€” no seed data for this service). Two global exception handlers as safety nets. `GET /health` returns service name, environment, and status. Runs on port `8002`.

## Core Domain Concepts

**Payment** â€” A single payment attempt for a commerce checkout. One checkout maps to one payment in the happy path. A failed payment may result in a new payment attempt (new `Payment` record, same `checkout_id`). Carries the external provider's reference ID once the provider acknowledges the request.

**Refund** â€” A request to reverse some or all of a captured payment. One payment may have multiple partial refunds. The sum of all refund amounts must not exceed the captured payment amount â€” enforced in the service layer.

**PaymentStatus transitions:**
`INITIATED â†’ PENDING â†’ CAPTURED` (success path)
`INITIATED â†’ PENDING â†’ FAILED` (provider rejection)
`CAPTURED â†’ REFUND_INITIATED â†’ REFUNDED` (refund path)
`INITIATED / PENDING â†’ CANCELLED` (timeout / checkout expiry)

**RefundStatus transitions:**
`INITIATED â†’ PROCESSED` (success)
`INITIATED â†’ FAILED` (provider rejection)

**PaymentMethod values:** `UPI`, `CARD`, `NET_BANKING`, `WALLET`

**Currency values:** `INR`, `USD`, `GBP`, `EUR`, `JPY`

## Integration Points

**Commerce service (event-driven, planned)**
Commerce publishes `CheckoutPaymentRequested` when a checkout is ready to be paid. Payment consumes this event to call `payment_service.initiate_payment()`. Payment publishes `PaymentCaptured` (triggers commerce to call `commit_reservation` and confirm orders) and `PaymentFailed` (triggers commerce to release inventory and cancel orders). These events are not yet implemented â€” the current `POST /payments/initiate` API serves as the synchronous stand-in.

**External payment providers (planned)**
Razorpay or Stripe integration will live inside the service layer (not the route layer). The service will call the provider SDK, store the provider reference ID, and update status based on the response. Provider calls happen inside the same DB transaction as the `Payment` record creation when possible (write-ahead: create the record, then call the provider, then update).

**PostgreSQL (primary store)**
Same `ledgerflow` database as the commerce service. The payment service owns the `payments` and `refunds` tables. Cross-service data (e.g. `checkout_id`) is stored as a plain `VARCHAR(36)` with no database-level FK constraint, since the referenced rows live in commerce's schema.

## Important Architectural Decisions

**Cross-service foreign keys stored as plain strings**
`Payment.checkout_id` references a checkout that lives in the commerce service's schema. There is no database-level FK constraint â€” enforcement is at the application level (commerce creates the checkout before publishing the event; payment trusts the event payload). This avoids cross-schema FK constraints and the coupling they would create.

**Single DB transaction per payment operation**
Each payment operation (initiate, capture, fail) is a single atomic transaction: create / update the payment record, update status, commit once. No multi-step saga within a single service.

**No seeders**
The payment service has no pre-seeded data. Unlike commerce (which seeds users and products for development), all payment records originate from live flows. The `seeders/` directory exists for structural consistency and future use.

**`services/` not `service/`**
The business logic directory is named `services/` (plural). Do not rename it to match commerce's `service/` â€” keep the existing folder name.

**Synchronous SQLAlchemy**
Same as commerce: sync SQLAlchemy with `psycopg2-binary`. Deliberate simplicity tradeoff at this development stage.

## Known Constraints

- **No domain models yet.** `models/__init__.py` is empty. Running migrations now applies no schema changes. Add `Payment` and `Refund` models, then create `0001_initial_schema.py`.
- **No routes yet.** `main.py` only serves `GET /health`. No payment or refund APIs are registered.
- **No event consumption.** Kafka / event bus infrastructure does not exist. The payment initiation flow is driven by direct HTTP calls for now.
- **No provider integration.** External provider calls are not implemented. Service methods will log a placeholder in the same position as commerce's payment placeholder.
- **`onupdate` is ORM-only.** `TimestampMixin.updated_at` via `onupdate=func.now()`. Raw SQL `UPDATE` statements must manually include `updated_at = now()`.
- **Enum drift risk.** PostgreSQL enum types will be defined in three places: migration SQL, `PGEnum` instances in migration, and Python enum classes in `utils/enums.py`. All three must be updated together when enum values change.
- **`services/` vs `service/`.** This service uses `services/` (plural). Do not change this to match commerce.

## Development Notes

**Adding the first domain model**
1. Create `models/payment.py` with the `Payment` ORM class inheriting from `Base` and `TimestampMixin`.
2. Add it to `models/__init__.py` via `from models.payment import Payment`.
3. Run `alembic revision --autogenerate -m "0001_initial_schema"` from `services/payment-service/`.
4. Review the generated migration â€” replace `sa.Enum` with `PGEnum(..., create_type=False)` after manually adding `op.execute("CREATE TYPE payment_status AS ENUM (...)")` in `upgrade()` and the corresponding `DROP TYPE` in `downgrade()`.

**Adding a new enum value**
Do not edit existing migration files. Create a new migration: `alembic revision -m "add_value_to_payment_status"`. Then add the value to `utils/enums.py`.

**Error handling convention**
Three-layer exception handling â€” identical to commerce:
1. **Repository** â€” catches `SQLAlchemyError`, re-raises as `DatabaseException`. Domain failures raise the appropriate `AppException` subclass directly.
2. **Service** â€” catches `AppException` subclasses and re-raises unchanged. Catches bare `Exception` and wraps as `ServiceException`. Never swallows silently.
3. **Routes** â€” catches `AppException` â†’ `JSONResponse(status_code=exc.status_code, content=ErrorResponse.from_exception(exc).model_dump())`. Catches bare `Exception` â†’ `JSONResponse(500, ErrorResponse.internal_error().model_dump())`.
4. **Global handlers in main.py** â€” safety net only.

**Response convention**
All route handlers return `SuccessResponse.ok(data=...)` for reads and `SuccessResponse.created(data=...)` for creates.

**Session management**
Never hold a `Session` outside a repository method or middleware dispatch block. Session provided by `get_db()` FastAPI dependency, closed after each request. Services and repos must not store the session on `self`.

**Instantiation**
Repos instantiated in the `__init__` of the service that owns them. Services instantiated at the top of each route handler function. Neither repos nor services have module-level singleton instances.

## AI Agent Context

**Rules an AI agent must not break without understanding the wider impact:**
